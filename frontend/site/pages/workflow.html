A new way of working

Manifest doesn't just help you write code. It changes how you maintain, update, and evolve a codebase — with agents that understand both your application and its relationship to the framework.

You clone Manifest. You build your app — features, schemas, services. Weeks pass. You add authentication, wire up a database, build an API that your frontend talks to. The framework source lives right there in manifest/, and you've read it. Maybe you've even tweaked a few things.

Meanwhile, upstream evolves. New CLI commands. Better validation. Improved error handling. The framework keeps getting better — and your project has diverged from it in ways that matter.

In any other framework, this is where things get painful. In Manifest, this is where agents earn their keep.

$ manifest update

You say update. The agent understands both sides.

▸ fetching manifest-upstream... 5 new commits
▸ reading commit messages... grouping by area
▸ batch 1: CLI improvements (3 commits)... recommend: pick
▸ batch 2: example feature (2 commits)... recommend: skip
▸ cherry-picking batch 1 onto main... clean
▸ bun test / tsc / manifest check all pass

This isn't npm update. There's no black-box package to upgrade. The framework is source code in your project — and the agent reads every upstream commit, groups them by area, and recommends what to cherry-pick and what to skip. You decide what enters your app.

The two-branch model makes this clean. The manifest branch tracks upstream. Your main branch is your app. Updates flow one direction: upstream → manifest → cherry-pick onto main. When there's a conflict, the agent categorizes the file — framework improvement or application code — and makes the right call. Your features are sovereign. Framework improvements are welcome. The agent knows the difference.

$ manifest commit

Every commit is a letter to the next agent

Before — what most projects get:
fix stuff
After — what Manifest expects:
fix(registration): handle null displayName from OAuth providers
OAuth providers can omit the display name field entirely.
The registration feature assumed it was always present
because the input schema marks it required — but OAuth
registrations bypass input validation.
Root cause: features/UserRegistration.ts line 34
accessed input.displayName without a null check.

In Manifest, commits aren't an afterthought — they're infrastructure. The commit history is how agents orient themselves when they arrive at your codebase. It's how manifest update knows what to accept, skip, or adapt. It's how debugging works without git bisect.

A commit message is a letter to the next agent explaining what you did and why. What changed. Why it changed. The approach taken. What was deliberately left out. When the next agent reads git log, it should understand the progression without looking at a single diff.

$ manifest learn

After big changes, nothing drifts

▸ checking MANIFEST.md... stale — 2 new features not indexed
▸ checking AGENTS.md... accurate
▸ checking skills... all current
▸ checking types... tsc clean
▸ checking tests... 1 feature missing test file
▸ checking conventions... manifest check clean

You added three features, installed an extension, and modified a service. The code works. But does the rest of the codebase know? Is MANIFEST.md still accurate? Do the skills still reflect reality? Are there test files for everything?

manifest learn treats your codebase as a living system. When one part changes, the agent checks whether other parts need to catch up — documentation, indexes, types, tests, conventions. The next agent that arrives should never follow stale instructions because you forgot to update the map after moving the furniture.

Framework improvements get accepted. Your features stay untouched. Shared files like AGENTS.md get both sides — upstream's wording polish and your architecture notes. And VISION.md? That's your app's soul. Upstream has no say. The agent will never change what your application is supposed to be.
