What is Manifest

An experiment in framework design. The premise: as agents take on more of the coding, the frameworks they work with could be designed around how they read and reason — not just how humans do.

Every framework you've used was designed for human readers. Documentation, conventions, abstractions — optimized for the way people think about code. That works. But agents process code differently. What if the foundation accounted for that?

No dependencies. Nothing hidden behind node_modules.
No hidden functionality. No magic methods, no lifecycle hooks.
No docs needed. What you read in code is what you get.
The idea: an agent reads it and builds. No framework knowledge required.

Whether that trade-off is worth it is something you'd have to try and decide for yourself.

How it works

Manifest isn't a dependency you install — it's source code you own. The entire framework lives in your project. You can read every line, modify anything, and your agent can too.

Setup is agent-driven. Paste the Spark prompt, describe what you want to build, and the agent handles the scaffolding — cloning the repo, setting up branches, writing your VISION.md, getting the server running.

// you say: "a recipe app for meal planning"
▸ cloning manifest... done
▸ setting up branches... manifest ↔ main
▸ renaming to meal-planner... done
▸ writing VISION.md... done
▸ bun --hot index.ts server running on :8080

When you're ready to deploy, docker compose up --build works out of the box. No build pipeline to configure, no platform lock-in.

Your code. Your repo. Your infrastructure. Nothing opaque between you and what's running.

agent skills — not CLI commands

What if your agent could operate the framework?

Manifest isn't just code your agent can read — it's designed so your agent can operate it. Some things are CLI commands. Others are skills — structured workflows your agent loads and executes when you ask.

Say "update from upstream" and the agent loads the update skill, fetches upstream, reads every commit, and walks you through what to pick.

› update from upstream
loading skill: manifest-update
▸ fetching manifest-upstream... 3 new commits
▸ reading commit messages... understood intent
▸ recommending: pick 2, skip 1... your call
› pick both
▸ cherry-picking onto main... 1 conflict
▸ resolving server.ts... framework → accept
▸ running tests... all passing

Updates, commits, conflict resolution — these are agent skills, not shell commands. The agent handles the workflow. You handle the decisions.

Meet Spark

If agents can read the code, can they also watch it run?

Spark is a sidekick agent that runs alongside your application. When something goes wrong — an error, a crash, an unhandled exception — it sees it with full context and can investigate, diagnose, and fix.

// spark session — live
▸ environment: production · mode: fix
▸ watching: .spark/events/
▸ event: 500 in user-registration — reading feature file...
▸ fixed: null check on displayName · hot-reloaded · 4.2s

Every request is a structured JSON envelope with status, context, and trace ID. Every error is a JSON file. Always inspectable. By you. By the agent. By anything.

This is the part of the experiment we're most curious about. An agent that doesn't just build your app but monitors it — in development and in production. It works surprisingly well for some things. For others, it's still early. That's kind of the point.
